Hexo 搭建博客踩坑记录

发表于 2023-05-21 更新于 2024-08-10 阅读次数：
使用 Hexo 在 GitHub Pages 搭建博客的踩坑记录与经验，作为参考。官方文档已经介绍很多了，文章的内容也是基本按照官方文档的架构来的。这里主要是补充一些别的内容。

概述

https://hexo.io/zh-cn/docs/

里面说到了前置准备的内容。此外，需要注意的是，要将自己的 SSH key 添加到自己的 GitHub 账户中，之后的仓库推送等操作，要用到仓库的 SSH 地址，便于免密验证。

在自己 GitHub 中要有一个 public 仓库，名字为你的 GitHub 用户名.github.io，用于部署。可以先看这部分内容：https://hexo.io/zh-cn/docs/github-pages，但是先看最开头的部分就好。

建议使用 VS Code，方便查看文件目录，修改文件。

安装时，指定的文件夹，是作为自己博客的目录。文件夹的名字没有特别的要求，遵循一般的文件命名规范和习惯即可。

仓库备份

官方文档没怎么说到仓库备份的内容。这里就具体说一下。之前官方文档说了，每次部署上去之后，这个仓库存储的并不是网站根目录文件夹，而是.deploy_git文件夹，这个文件夹是动态生成的，每次部署时的内容都不一样。那么，要备份自己的仓库的内容，就要在 GitHub 再新建一个仓库，再用 Git 进行配置与同步。

上述配置命令成功之后，以后每次要备份，就是常规操作了。

主题

https://hexo.io/zh-cn/docs/themes

我的评价是，这篇的内容，对于新人没什么用，对于熟练者又大概是用不上，反正他们早就会使用甚至自己制作主题了。

https://hexo.io/themes/

有许多不错的主题可用，这里只是官方推荐的主题。但实际上很多大佬自己开发设计了不错的主题，可以自行搜索。如果是初次使用 Hexo，非常建议使用 NexT 主题，体验极好。不过到了后面，可能会改为使用更适合自己品味的主题。

设置主题的方式是，修改根目录_config.yml的theme的值，为相应主题的名字。

大多数主题的传统安装方法是：将主题相应的 GitHub 仓库 clone 到themes文件夹中。每个主题都有自己的配置文件_config.yml，位于自己主题的文件夹中。

但是，注意到，自 Hexo 5.0.0 起，增加了个特性，可以设置独立的_config.[theme].yml文件，放于站点根目录下，也就是与_config.yml同级的位置。这里的[theme]就是相应的主题的名字，比如 NexT 主题的配置文件就是_config.next.yml。换句话说，每个主题的配置现在可以在网站根目录下独立设置了。主要看“代替主题配置文件”的部分。

https://hexo.io/zh-cn/docs/configuration

这样的好处在于，你可以在本地安装多个主题，能够很方便地保存各个主题的配置，并且很方便地切换，只要修改_config.yml的theme的值即可。

另一个好处就是，可以方便如今很多主题支持的另一种安装方式：通过 npm 安装。这样，主题就会安装在node_modules目录下。如果不使用上述特性，那么你要在node_modules目录下的相应主题插件文件夹的_config.yml修改，翻找很麻烦。而使用了_config.[theme].yml特性，可以很方便地进行主题的配置。只要将相应主题的配置文件复制到网站根目录下，文件名修改为上述格式即可。

另外，经过测试，对于使用 npm 安装的主题，如果不使用_config.[theme].yml特性，似乎远程部署会失败。在许多主题的教程中，也确实更推荐这种操作方法，这大概是一种发展的趋势。

无论是 GitHub 仓库 clone 还是 npm 安装，每个主题本身是会有版本更新的，相应的_config.yml也会更新，这样，如果之前在里面自己修改了配置，那么更新后可能会覆盖。目前的解决办法有：自己 fork 那个主题的仓库到自己的 GitHub，或者是使用_config.[theme].yml来存，然后更新时手动进行合并。
