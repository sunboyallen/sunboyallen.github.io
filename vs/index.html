Windows上使用Visual Studio进行C++项目开发，调用第三方库

发表于 2024-03-12 更新于 2024-06-29 阅读次数：
如题。

写在开头

我之前在博客曾写过一篇文章，关于如何在Windows系统使用VS Code写C++代码调用OpenCV库。也包括以及遗留的问题，主要是依然没做到静态编译，也就是编译生成的exe没法单独跑，得加各种dll文件。以前一直嫌弃VS笨重，然而当我最终还是在VS上编译了OpenCV酷，调用并且确实实现了静态编译之后，我就把那篇文章删掉了，没必要了。

能看到群友大佬调侃说，现在的高级语言，如果包管理比C++还差，都不好意思叫作高级语言。关于这方面的事情，有很多历史原因，你乎也有很多内容，不赘述。主要想说，像这种在Windows上的项目级别的C++开发，特别是要用第三方库的，还得是Visual Studio。

说起来，要使用OpenCV，使用Python显然是更好的选择。再不济，在Linux上搞也还行。如果是自己在Windows上使用C++调用，那其实VS Code也够用：CMake先编译，配完加环境变量，跑还是能跑的，无非是如果exe单独跑的话还得加各种杂七杂八的dll动态库文件。

但如果是项目级的，这样显然是不够的，如果需要exe能单独跑呢？关键是如果只是缺少OpenCV相关的dll还好，你不静态编译，指不定别人电脑还缺少什么别的dll，这也体现出了Windows上的VS Code在面对C++包管理（第三方库）的时候难以解决的静态编译问题。

CMake

好就好在现在的第三方库的编译基本都有提供CMake的方式。使用cmake-gui，将CMakelists.txt文件所在的文件夹作为source code，当然直接把这个txt拖进去更方便。然后Configure，Generate，Open Project，生成，一路下来。

我把需要注意的点罗列在下面。

  • build的目录往往需要自己选择使用目录下的build文件夹。
  • Configure选择编译器的时候，选择Visual Studio提供的编译器，选自己电脑相应的版本。
  • 编译选项的勾选，对于有静态编译的需求的，记得勾选static相关选项，或者是取消勾选shared相关选项。
  • 在Visual Studio里生成的时候，注意选择好上面的Release/Debug，x86/x64，并且与后面自己的项目的选择相一致。

Visual Studio

之后的操作就很常规了，项目属性一通配置，网上一找一大把，我看基本够用了，我就不多说，只是自己补充一些。

  • 项目级的开发，我是比较推荐把第三方库直接拷贝一份在项目目录里的。这样，项目属性引入外部库，附加依赖项之类的，也直接用相对路径，比如$(ProjectDir)lib之类的写法，这样跨端开发能省去很多重复的操作。
  • 注意C/C++ -> 代码生成 -> 运行库，选择相匹配的/MT，/MTd，/MD，/MDd，根据实际需求，静态动态，是Debug还是Release决定。

这样一来，一个通用的，傻瓜式的流程就非常明确了，几乎适用于要用到任何第三方库的C++项目：

获取源代码 -> CMake编译 -> VS生成 -> VS项目配置

我自己试过了不少第三方库了，清一色基本是这个流程。有些库甚至不需要自己CMake生成，直接是头文件引用，那么配置就更方便了。我简单列出一些。

OpenCV

https://opencv.org/

https://github.com/opencv/opencv

现在opencv他提供的exe，是获取到源代码，头文件，库文件（包括静态库动态库），库文件默认是整合到一起的，相当于自己cmake的时候勾选了BUILD_OPENCV_WORLD，主要就是一个opencv_world的dll或者lib文件，不过好像依然不太行，还是不如自己下载源代码自己编译，可选择的选项也自由一些。

网上配置的教程比较多了，主要是这篇教程，详细说了Configure时候如果遇到下载超时的处理。手动下载，放到指定目录，有的还要MD5校验，加上MD5值。

LZ4

https://lz4.org/

https://github.com/lz4/lz4

直接clone过来，在build/cmake里。

zlib

https://zlib.net/

https://github.com/madler/zlib

可以官网点击下载zip，根目录就是了。

nlohmann JSON for Modern C++

https://json.nlohmann.me/

https://github.com/nlohmann/json

克隆下来，主要是用里面的include或者single_include文件夹（后者里面就一个nlohmann/json.hpp），添加到项目目录里就行了，不需要去CMake之类。

TinyXML2

https://github.com/leethomason/tinyxml2

克隆下来，根目录就是了。这个好像没有特定的静态编译选项，生成完直接用lib应该就行，项目配置时，代码生成，运行库选择带dll的相应的，最终效果也是exe能单独跑。

虽说也是基于C++标准库，不过我看网上的教程，如果不CMake，需要自己拷贝里面五六个文件到自己项目里，如果CMake了，最终在include里面就一个tinyxml2.h，再加上lib，个人感觉更方便。

Eigen

https://eigen.tuxfamily.org/

https://gitlab.com/libeigen/eigen

在release里下载，解压完根目录就是。

同样的这个CMake之后也没有lib，直接用include就行了。

gflags

https://gflags.github.io/gflags/

https://github.com/gflags/gflags

直接克隆，根目录就是。但是这个库配置过程有些个曲折，我专门写一篇来讲，可以看这篇。
